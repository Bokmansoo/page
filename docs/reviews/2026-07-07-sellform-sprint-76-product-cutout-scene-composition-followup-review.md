# Sprint 76 보완 코드리뷰 — 실제 상품 누끼 기반 장면 합성

## 결론

Sprint 76의 큰 방향은 구현되어 있었지만, 코드리뷰 결과 한 가지 핵심 누락이 있었습니다.

- 실제 상품 누끼 생성/저장/메타데이터 모델링: 구현됨
- 이미지 후보 API의 누끼 메타데이터 반환: 구현됨
- 기획 초안 승인 후 이미지 생성 작업에서 상품 누끼를 기준 이미지로 연결: 보완 전 누락, 보완 완료
- 프론트 후보 패널의 누끼/합성/검수 필요 상태 표시: 보완 완료

## 발견한 문제

`approve_planning_draft`에서 `ImageGenerationJobRecord`를 만들 때 `source_asset_ids=[]`, `preserve_product_identity=False`로 저장되고 있었습니다.

이 상태에서는 사용자가 업로드한 실제 상품 이미지에서 누끼를 만들어도, 기획 초안 승인 후 생성되는 이미지 작업이 그 누끼를 기준 이미지로 사용하지 못합니다. 즉, Sprint 76의 핵심 목표인 “실제 상품 외형을 유지한 장면 합성”이 결과 생성 단계까지 이어지지 않았습니다.

## 보완한 내용

1. 이미지 생성 작업에 상품 기준 이미지를 연결했습니다.
   - 프로젝트의 누끼 자산을 우선 사용합니다.
   - 누끼 자산이 없으면 업로드/URL 추출 원본 이미지를 fallback 기준 이미지로 사용합니다.
   - hero, lifestyle, detail, features, cta 계열 섹션에만 기준 이미지를 연결합니다.
   - specs/comparison/pre_purchase 같은 정보형 섹션은 기존 기획대로 HTML 그래픽 중심을 유지합니다.

2. 이미지 생성 프롬프트를 보강했습니다.
   - 제공된 상품 누끼/기준 이미지를 고정된 상품 정체성으로 사용하도록 명시했습니다.
   - 상품 형태, 색상, 비율, 로고/디스플레이, 눈에 보이는 특징을 유지하도록 제한했습니다.
   - 배경, 조명, 그림자, 장면만 합성하도록 지시했습니다.
   - 이미지 안에 텍스트, 로고 오버레이, 워터마크, 배지를 넣지 않도록 제한했습니다.

3. 상품 누끼 서비스 회귀 테스트를 추가했습니다.
   - 흰 배경이 투명 처리되는지 확인합니다.
   - 생성된 누끼 자산이 원본 `source_asset_id`를 유지하는지 확인합니다.
   - `cutout_status`, `background_removed`, `product_identity_preserved` 메타데이터가 저장되는지 확인합니다.

4. 기획 초안 승인 회귀 테스트를 추가했습니다.
   - hero 이미지 생성 작업이 실제 누끼 자산 ID를 `source_asset_ids`로 받는지 확인합니다.
   - `preserve_product_identity=True`로 저장되는지 확인합니다.
   - specs 섹션은 이미지 생성 작업 없이 `html_graphic`으로 남는지 확인합니다.

5. 결과 화면의 이미지 후보 패널 표시를 보완했습니다.
   - 실제 상품 누끼 사용
   - AI 배경 합성
   - AI 보정 후보 - 검수 필요
   - 상품 형태 변경 가능성 있음

## 빌드 참고

`npm.cmd run build`는 코드 에러가 아니라 Windows/Next 빌드 프로세스가 종료되지 않고 남는 현상이 반복되어 이번 Sprint 76 완료 판정에서는 제외했습니다. 남아 있던 `next build` 프로세스는 종료했습니다.

전체 빌드 안정화는 별도 환경/빌드 프로세스 이슈로 분리해 다루는 것이 좋습니다.

## 판정

Sprint 76은 보완 후 기획 의도에 맞게 구현된 상태로 판단합니다.

다만 실제 이미지 품질은 mock이 아니라 실제 이미지 생성 API 모드에서 한 번 더 확인해야 합니다. 특히 “정면 상품 사진을 장면 안에서 손에 쥔 것처럼 자연스럽게 바꾸는 수준”은 단순 누끼 합성만으로는 한계가 있으므로, 다음 단계에서는 이미지 생성 모델에 기준 이미지와 장면 지시를 함께 전달하는 실제 모드 QA가 필요합니다.
