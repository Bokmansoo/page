# 리뷰 운영 가이드

셀폼의 설계, 기능, 코드, AI 워크플로 변경은 구현 또는 배포 전에 리뷰한다. 리뷰의 목적은 승인 자체가 아니라, 사용자 신뢰를 해칠 실패를 변경 전에 발견하고 근거 있는 결정을 남기는 것이다.

## 언제 리뷰하는가

- 제품 기획·아키텍처·데이터 모델을 확정하거나 크게 바꿀 때
- 사용자 흐름, AI 프롬프트·모델·스키마, 카테고리 규칙을 바꿀 때
- 외부 링크·파일 처리, 권한, 판매처 출력처럼 보안·정책 영향이 있는 기능을 추가할 때
- 배포 전 또는 장애·품질 이슈를 해결한 뒤

스프린트가 끝날 때는 반드시 코드 리뷰 기록을 `YYYY-MM-DD-기능명-sprint-N-code-review.md` 형식으로 남긴다. 코드가 아직 없는 Sprint 0은 조사·결정 리뷰로 대체한다.

## 리뷰 절차

1. 변경 목적과 범위를 [리뷰 기록 템플릿](TEMPLATE.md)에 작성한다.
2. 변경 내용을 읽고, 관련 기획·결정·테스트·조사 문서를 연결한다.
3. 아래 체크리스트를 기준으로 이슈를 기록한다.
4. 이슈는 심각도와 처리 상태를 정하고, 해결·보류·제외 사유를 남긴다.
5. 테스트 증적을 확인한 뒤 `승인`, `조건부 승인`, `반려` 중 하나를 결론으로 남긴다.
6. 발견한 반복 교훈은 `memory/`에, 장기 결정은 `docs/decisions/`에 옮긴다.

## 코드 리뷰 문서의 필수 구성

실제 코드 리뷰는 [리뷰 기록 템플릿](TEMPLATE.md)을 복사해 다음 순서로 작성한다.

1. 브랜치, 날짜, 범위, 리뷰어가 담긴 메타데이터 표
2. 변경 요약과 이번 변경이 필요한 이유
3. 심각도·위치·영향·제안·조치 상태를 가진 이슈 목록
4. 머지 또는 다음 스프린트 전 처리해야 할 우선순위 권고
5. 잘된 설계와 재사용할 만한 패턴
6. 테스트·검증 증적, 최종 결론, 남은 위험

조치가 끝난 이슈도 지우지 않는다. 원래 문제와 해결 내용, 확인 날짜를 남겨야 같은 문제가 재발했을 때 추적할 수 있다.

## 심각도

| 등급 | 의미 | 출시 기준 |
|---|---|---|
| 차단 | 데이터 유출·손상, 확인되지 않은 판매 사실 출력, 핵심 흐름 중단 | 해결 전 승인 불가 |
| 높음 | 결과 품질·신뢰·비용에 큰 영향, 명확한 우회가 없음 | 해결 또는 명시적 보류 승인 필요 |
| 보통 | 특정 상황에서 사용자 혼란·재작업 발생 | 해결 계획과 담당·기한 필요 |
| 낮음 | 개선 가치가 있으나 현재 사용을 막지 않음 | 백로그 또는 후속 작업으로 기록 |

## 공통 체크리스트

### 제품·사용자 경험

- 사용자가 이 변경으로 얻는 결과가 한 문장으로 명확한가?
- 입력 없음, 느린 연결, 뒤로 가기, 중복 클릭, 실패 상태에서 다음 행동이 보이는가?
- 모바일 기준의 긴 상세페이지 편집·검수 경험이 유지되는가?

### 사실·AI 신뢰성

- 판매 문구가 `확인됨` 사실에 연결되는가?
- 미확인 사실은 최종 출력에서 제외되는가?
- AI 결과가 실패·빈 결과·형식 오류일 때 수동 대체 흐름이 있는가?
- 프롬프트·모델·스키마 변경의 품질·비용 영향이 평가됐는가?

### 보안·데이터

- 워크스페이스 권한과 파일 접근이 분리되는가?
- 외부 링크·파일 입력을 안전하게 검증하는가?
- 변경이 기존 출력물·사실·감사 이력을 덮어쓰지 않는가?
- 민감한 값과 공급처 원본이 로그·오류 화면에 노출되지 않는가?

### 품질·운영

- 자동 테스트, 수동 QA, 출력 스크린샷 중 필요한 증적이 있는가?
- 오류 코드, 재시도, 사용자 안내, 운영 알림이 정의됐는가?
- 작업 시간·AI 비용·렌더링 영향이 허용 범위인가?
- 롤백 또는 이전 버전 복원 방법이 있는가?

## 결론 기준

- **승인:** 차단·높음 이슈가 없고 검증 증적이 충분하다.
- **조건부 승인:** 남은 이슈의 소유자·기한·안전한 우회책이 명확하다.
- **반려:** 핵심 흐름, 사실 신뢰성, 보안, 데이터 무결성 중 하나가 불명확하다.
