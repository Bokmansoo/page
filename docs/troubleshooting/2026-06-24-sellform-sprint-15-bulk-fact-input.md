# 트러블슈팅: Sellform Sprint 15 Bulk Fact Input UX

## 1. 개요

일괄 사실 추가 기능 개발 및 프론트엔드 빌드 최적화 과정에서 감지된 문제들과 대응 방침, 후속 조치 사항들을 수록합니다.

## 2. 직면한 기술 이슈 및 조치 사항

### M1. TypeScript strict type-check 에러 (no-explicit-any)

- **증상**: Next.js 14 Web App 프로덕션 빌드(`npm.cmd run build`)를 구동할 때, eslint 린팅 검사 단계에서 실패했습니다.
- **원인**:
  - `BulkFactModal.tsx` 컴포넌트의 콜백 인터페이스 `onSuccess` 파라미터 타입으로 `any[]`를 명시적으로 사용했기 때문에, strict typescript lint 규칙(`@typescript-eslint/no-explicit-any`) 위반 오류가 검출되었습니다.
- **조치**:
  - 모달 파일 내부에 백엔드 API가 리턴하는 JSON 사실 스키마와 1대1 매핑되는 `BulkFactResponse` 타입을 정의했습니다.
  - `onSuccess` 함수의 createdFacts 파라미터를 `BulkFactResponse[]`로 변경하여 린팅 컴파일을 정상화하였습니다.

### M2. 50개 제한을 초과하는 대용량 일괄 입력 처리 시 서버 메모리 부하 위험

- **증상**: 사용자가 수백 개의 상품 사양이나 공급처 줄글을 하나의 입력창에 복사해서 일괄 저장을 구동할 경우 데이터베이스 커밋 과부하가 발생할 수 있습니다.
- **원인**: 일괄 트랜잭션의 적재 범위가 정해져 있지 않은 경우 서버 랙 및 타임아웃 오류를 유발합니다.
- **조치**:
  - 백엔드 `bulk_create_facts` 및 프론트엔드 모달 유효성 검사부에서 아이템 리스트 개수를 `1개 이상 50개 이하`로 제한하여 이를 초과할 시 즉시 alert 경고창 노출 및 HTTP 400 Bad Request 에러로 차단하도록 방어 코드를 신설했습니다.

### M3. 텍스트 내 중복 공백 및 정규화 편차에 의한 중복 확인 누락

- **증상**: `사실 1`과 `사실 1 `(끝에 공백 포함) 혹은 줄바꿈 정규화 차이에 의해 중복 등록이 제대로 걸러지지 않는 경우.
- **원인**: 단순 `fact_text` 문자열 단순 동등비교(`==`) 시 공백 문자 차이로 인해 중복 배제가 생략되었습니다.
- **조치**:
  - 백엔드에 기존 구축되어 있던 `normalize_fact_text` 헬퍼 메서드를 활용하여, 특수문자와 앞뒤 공백을 정규화한 문자열 기준(`existing_normalized`)으로 동등성을 조회하도록 중복 필터링을 고도화했습니다.

## 3. 향후 개선 사항

- **엑셀/CSV 업로드 연동**: 단순 줄바꿈 복사-붙여넣기뿐 아니라, 소싱 자료를 담은 스프레드시트 템플릿(Excel, CSV)을 드래그 앤 드롭하여 대량의 팩트 카드를 임포트해오는 편의 장치 제공이 필요합니다.
- **일괄 상태 변경 제어**: 수동 생성뿐 아니라, 체크박스로 여러 팩트 카드를 일괄 선택하여 승인(`confirmed`) 또는 수정 필요(`needs_revision`) 상태를 벌크로 토글할 수 있는 UX가 요구됩니다.

## 4. 결론

- strict typescript 컴파일 방해 요소를 해소하고 안정장치를 강화하여, 일괄 입력 UX가 안정적으로 배포될 수 있는 높은 완성도를 확보했습니다.
