# 트러블슈팅: Sellform Sprint 12 Image OCR

## 1. 개요
이미지 텍스트 분석(OCR) 파이프라인 구축 및 API 연동 과정에서 나타난 분석 실패 케이스에 대한 우회 처리 및 장기 과제 기록입니다.

## 2. 발견 이슈 및 해결 조치

### M1. 이미지 텍스트 미검출 시 E2E 테스트 오작동 위험 (Fallback 대응)
- **증상**: 이미지에서 텍스트가 추출되지 않아 소스 수집 리스트(`CollectedSource`)에서 자산을 아예 제외할 경우, 기존의 단순 에셋 연동 로직만 확인하는 `test_auto_extract_creates_image_asset_candidate` 회귀 테스트가 깨지는 문제 발생.
- **원인**: 기존 E2E 테스트는 텍스트 미검출 자산에 대해서도 기본 candidate를 만들어 `source_asset_id` 매핑이 되었는지를 엄격히 검증하고 있었음.
- **조치**: 
  - 이미지에서 텍스트가 분석되지 않았더라도, 기존 API 규격 준수용 Fallback 데이터(`Uploaded image asset: {filename}`)를 CollectedSource에 수집하여 API가 정상 완료되게 함.
  - 동시에, `failed_sources` 배열에 분석 실패 사유(`image_text_unavailable`)를 기록하여 프론트엔드가 이를 인지하고 사용자에게 경고 배너를 노출하도록 수정하여 문제 해결.

### M2. OCR 분석 후보 신뢰도 한계 적용
- **증상**: 이미지 분석을 통해 얻은 사양이 검수 과정 없이 상세페이지 제작에 자동 확정 기재될 위험이 존재함.
- **원인**: Mock OCR 텍스트 역시 일반 설명글처럼 수집되기 때문에, 매칭되는 팩트들이 높은 신뢰도로 인식되어 자동 승인 처리될 소지가 있음.
- **조치**: 이미지에서 유입된 팩트의 경우 `confidence = min(candidate.confidence, 0.72)` 상한을 엄격하게 적용하여 0.72 이하로 강제 억제하고, 무조건 `needs_review = True`로 세팅하여 사실 확인 보드에서 사람에 의한 검토 게이트를 통과하도록 조치.

## 3. 후속 과제

- **실제 클라우드 OCR / 멀티모달 LLM 연동**:
  - 현재는 파일명을 기반으로 작동하는 `MockImageTextExtractor` 구조입니다.
  - 다음 고도화 단계에서 Google Cloud Vision API 또는 OpenAI GPT-4o 멀티모달 API를 `ImageTextExtractor` 인터페이스 구현체로 추가하고, 설정 플래그에 따라 주입하는 방식으로 실제 기능을 마이그레이션해야 합니다.
- **보안 및 규정 동의 게이트**:
  - 민감한 공급처 정보나 사유 자산 이미지를 외부 클라우드로 업로드하기 전, 유저 설정 페이지 혹은 팝업을 통해 "외부 AI 이미지 분석 전송 동의"를 명시적으로 얻는 게이트웨이를 설계해야 합니다.

## 4. 결론
시스템 E2E 안정성 및 하위 호환성을 유지한 채, 이미지로부터 구조화된 상품 사실 후보를 생성할 수 있는 아키텍처적 초석을 성공적으로 안착시켰습니다.
