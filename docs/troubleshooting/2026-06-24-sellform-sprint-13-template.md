# 트러블슈팅: Sellform Sprint 13 Problem-Solution Template

## 1. 개요

문제 해결형 상세페이지 내러티브 템플릿(`problem_solution`) 개발 및 테스트 연동 과정에서 발생할 수 있는 주요 예외 상황, 런타임 오류의 예방 지침 및 후속 개선 과제를 정리합니다.

## 2. 발견 및 잠재적 이슈 분석

### M1. 문제 해결형 템플릿 구조 재생성 시 기존 버전 롤백 데이터 유실 우려

- **증상**: 사용자가 상세페이지 구조를 '카테고리 기본형'에서 '문제 해결형'으로 변경하여 "구조 재생성"을 호출할 경우, 기존에 편집했던 카피 텍스트가 모두 덮어씌워져 유실될 수 있습니다.
- **원인**: 백엔드의 `create_page_draft` API는 Overwrite 정책에 따라 기존 `ProductPage`와 그 하위 `PageSection`을 물리적으로 삭제(delete)하고 새로 생성(insert)합니다.
- **해결책/조치**: 
  1. 백엔드 `create_page_draft` 로직 호출 전, 기존 페이지 상태가 존재하면 이를 `PageVersion`에 자동으로 백업 스냅샷으로 적재한 후에 삭제를 실행하도록 안전 장치를 도입했습니다.
  2. 사용자는 유실된 이전 데이터가 필요할 경우 좌측 "자동 저장 버전 (롤백)" 패널을 통해 원하는 버전으로 복원(restore)할 수 있습니다.

### M2. 팩트 카드 수량이 부족한 경우 (`fact_ids < 2`) Index Out of Range 위험

- **증상**: 상품 등록 초기 단계이거나 확인된 상품 사실 카드(Confirmed Fact)가 1개 이하인 상태에서 `problem_solution` mock 템플릿 생성을 실행할 때, 백엔드 Python 코드 상에서 `fact_ids[1:2]` 또는 `fact_ids[:2]` 슬라이싱 시 비정상 배열 참조 에러가 일어날 위험이 있습니다.
- **원인**: mock generator 헬퍼(`_get_problem_solution_mock_page`)에서 팩트 목록 슬라이싱 인덱스를 세밀히 제어하지 않을 경우 발생할 수 있습니다.
- **해결책/조치**:
  - Python의 List Slicing (`list[start:end]`)은 배열 크기가 작아도 `IndexError`를 발생시키지 않고 빈 리스트 또는 존재하는 원소만 안전하게 반환합니다. 
  - 다만 `associated_fact_ids` 매핑의 무결성을 위해, `fact_ids`가 없더라도 에러 없이 뼈대 텍스트가 생성되도록 `fact_summary` 문자열의 삼항식 예외 처리(`if confirmed_facts else "확인된 상품 정보를 기준으로 구성했습니다."`)를 선제 적용하였습니다.

### M3. 미확정 사실의 카피 본문 오염 위험

- **증상**: LLM 호출 시 시스템이 '미확정 사실은 사용 금지' 프롬프트 지시를 무시하고 카피 문구에 혼입하는 현상.
- **원인**: LLM의 지시 불이행(Instruction Compliance) 편차.
- **해결책/조치**:
  - `system_prompt` 및 Pydantic tool 스키마의 `associated_fact_ids` 설명을 고도화하여 Claude 3.5 Sonnet이 오직 전달된 `confirmed_facts` 리스트의 팩트만 사용하도록 바인딩 제어를 가했습니다.
  - 프론트엔드 모바일 렌더링 영역 상단에 미확정 경고 배너(`warnings`)를 배치하여, 혹여나 미확정 사실이 혼용된 경우 퍼블리싱 전 직관적으로 리뷰 및 수정할 수 있게 시각적 유효성 검사 장치를 제공합니다.

## 3. 후속 고도화 과제

상세페이지 마케팅 도달률을 높이기 위해 차기 스프린트에서 다음과 같은 내러티브 템플릿의 추가 개발이 필요합니다.

- **스펙 비교형 템플릿**: 타사 대비 성능 우위를 표(Table)와 다이어그램으로 강조하는 이성적 설득 템플릿.
- **감성 브랜딩형 템플릿**: 브랜드 가치와 친환경 스토리텔링을 긴 줄글과 인물 중심 비주얼 섹션으로 배치하는 감성형 템플릿.
- **사용 장면형 템플릿**: 상황별 이용 후기 및 실제 일상 적용 모습을 스케치 형태로 나열하는 체험 중심 템플릿.

## 4. 결론

- 이번 문제 해결형 내러티브 템플릿 추가는 다양한 상세페이지 설득 전략의 토대가 될 것입니다. 잠재적인 에러 방지 패턴을 통해 런타임 신뢰성이 우수하게 유지되고 있음을 보증합니다.
# 보완 조치 기록 (2026-06-24)

## 해결된 이슈

### M1. 카테고리별 소구점이 범용 문구로 고정됨

- 원인: 문제 제기 제목만 카테고리별 맵을 사용했다.
- 조치: `main_claim`과 `secondary_benefit`에도 카테고리별 카피를 적용했다.
- 재발 방지: Fashion과 Living의 메인 소구점이 서로 달라야 한다는 회귀 테스트를 추가했다.

### M2. 실제 LLM이 템플릿 순서를 어겨도 저장될 수 있음

- 원인: 프롬프트 지시만 있고 응답 후 구조 검증이 없었다.
- 조치: 7개 section type의 정확한 순서와 확인된 fact ID만 허용하는 서버 검증을 추가했다.
- 재발 방지: 순서가 틀린 가짜 LLM 응답이 fallback으로 전환되는 회귀 테스트를 추가했다.

### m1. 잘못된 템플릿 값이 기본형으로 조용히 처리됨

- 원인: `narrative_template`이 자유 문자열이었다.
- 조치: `Literal["category_default", "problem_solution"]`으로 요청 계약을 제한했다.
- 재발 방지: 지원하지 않는 템플릿이 422를 반환하는 테스트를 추가했다.
