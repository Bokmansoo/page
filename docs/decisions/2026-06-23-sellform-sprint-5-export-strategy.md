# 결정 기록: Sellform Sprint 5 Export 전략

| 항목 | 내용 |
| --- | --- |
| 일자 | 2026-06-23 |
| 상태 | 채택 |
| 범위 | 상세페이지 Export, 마켓 프리셋, 이미지 패키징, 다운로드 흐름 |

## 1. 결정

Sprint 5의 Export 기능은 다음 구조로 구현한다.

- 상세페이지 섹션 데이터를 HTML/CSS 기반 렌더링 입력으로 변환한다.
- Playwright 사용이 가능한 환경에서는 브라우저 렌더링을 우선 사용한다.
- 테스트/로컬/오프라인 환경에서는 Pillow 기반 mock/fallback 렌더링을 허용한다.
- 생성된 이미지 slice와 manifest를 ZIP 패키지로 묶어 다운로드 가능한 Asset으로 저장한다.
- Export 실행 상태는 `ExportJob` 모델에 기록한다.
- 쿠팡/스마트스토어용 출력 크기와 slice 정책은 마켓 프리셋으로 분리한다.

## 2. 배경

Sellform의 핵심 목적은 “상품 정보 → 상세페이지 생성 → 검수 → 마켓 업로드 가능한 산출물” 흐름을 만드는 것이다. Sprint 5는 이 중 실제 판매처에 올릴 수 있는 형태의 산출물을 만드는 첫 단계다.

자동 업로드는 후속 고도화로 미루고, 현재 Sprint에서는 사용자가 검수 가능한 이미지/ZIP 산출물을 안정적으로 만드는 것을 우선한다.

## 3. 선택 이유

- 브라우저 렌더링은 실제 상세페이지 레이아웃에 가까운 결과물을 만들 수 있다.
- fallback 렌더링은 CI, 로컬 테스트, 외부 의존성 없는 검증에 유리하다.
- ZIP 패키징은 쿠팡/스마트스토어 업로드 전 사람이 검수하기 쉽다.
- `ExportJob` 저장은 실패 원인, 진행 상태, 재시도 흐름을 추적하기 좋다.
- 마켓 프리셋 분리는 향후 쿠팡/스마트스토어 외 채널 확장에 유리하다.

## 4. 대안

### 대안 A. 처음부터 마켓 API 자동 업로드

초기 사용성은 좋아 보이지만, 각 판매처 정책/API/권한/심사 이슈에 막힐 가능성이 크다. 상세페이지 생성 도구의 핵심 품질을 검증하기 전에 외부 연동 리스크가 커진다.

### 대안 B. HTML만 Export

개발은 단순하지만 쿠팡/스마트스토어 실무에서는 이미지 기반 상세페이지가 더 직접적이다. 내부 사용 도구 단계에서는 이미지/ZIP 결과물이 더 현실적이다.

### 대안 C. 렌더링을 전부 mock으로 처리

테스트는 쉬워지지만 실제 산출물 품질을 확인하기 어렵다. 따라서 mock은 fallback으로만 둔다.

## 5. 결과와 영향

- Sprint 5는 상세페이지 이미지 산출과 ZIP 다운로드까지 완료된 상태를 목표로 한다.
- Export 이력과 실패 상태를 추적할 수 있어 운영/리뷰 문서 작성이 쉬워진다.
- 추후 Figma MCP, Google Drive, Notion, Browser QA, 마켓 API 자동화는 Export 결과물을 기준으로 플러그인처럼 붙일 수 있다.

## 6. 남은 결정

- 운영 전 쿠팡/스마트스토어 최신 이미지 정책을 공식 문서 기준으로 재검증한다.
- 실제 배포 환경에서는 local upload 저장소를 object storage 또는 CDN으로 교체할지 결정한다.
- 병렬 테스트 환경을 지원할 경우 worker별 DB/파일 저장소 격리 방식을 별도 결정한다.
