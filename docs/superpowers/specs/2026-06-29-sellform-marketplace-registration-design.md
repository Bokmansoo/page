# Sellform 마켓 등록 연동 설계

## 1. 문서 목적

Sellform에서 완성한 상세페이지와 상품 정보를 네이버 스마트스토어 및 쿠팡에 등록할 수 있도록 후속 제품 구조를 정의한다.

첫 버전의 목표는 무인 자동 게시가 아니다. Sellform이 마켓별 등록 초안을 만들고 필수값과 오류를 검사한 뒤, 사용자가 최종 내용을 확인하고 명시적으로 승인했을 때만 외부 마켓에 전송하는 것이다.

## 2. 결정 요약

- 상세페이지 제작과 마켓 등록을 하나의 긴 함수로 결합하지 않는다.
- Sellform 공통 상품 데이터와 마켓별 요청 형식을 분리한다.
- 마켓 등록 초안은 Sellform 내부에만 저장한다.
- 사용자 승인 전에는 외부 상품 생성 API를 호출하지 않는다.
- 네이버 스마트스토어 연동을 먼저 완성한 뒤 쿠팡으로 확장한다.
- 실제 전송은 멱등 키, 요청 스냅샷, 응답 원문, 재시도 이력을 남긴다.
- 마켓 API를 사용할 수 없거나 계정 연결이 안 된 사용자는 등록 패키지를 내려받아 수동 등록할 수 있다.

## 3. 선행 조건

다음 작업이 먼저 완료되어야 한다.

1. Sprint 35: Figma 시각형 상세페이지 렌더러
2. Sprint 36: 업로드 이미지 및 URL 수집 이미지의 섹션 자동 매핑
3. Sprint 37: 디자인과 판매 전략 후보 선택

마켓 등록 연동은 이미지 생성 품질을 개선하는 기능이 아니다. 이미 승인된 상품 데이터와 상세페이지 결과물을 마켓 규격으로 변환하고 전달하는 기능이다.

## 4. 사용자 흐름

```text
상세페이지 완성
  → 마켓 등록 준비
  → 공통 상품 정보 확인
  → 등록할 마켓 선택
  → 마켓 계정 연결 확인
  → 마켓별 필수값 자동 변환
  → 누락값·정책·이미지 규격 검증
  → 등록 미리보기
  → 사용자 최종 승인
  → 외부 마켓 API 전송
  → 등록 상태 조회
  → 성공 URL 저장 또는 실패 항목 수정·재시도
```

계정 연결이나 API 전송을 원하지 않는 사용자는 `마켓 등록 패키지 다운로드`를 선택할 수 있다.

- 실패 원인 표시와 안전한 재시도
- 요청·응답·승인·재시도 감사 로그

### 6.1 공통 상품 패키지

마켓별 API 필드를 페이지 컴포넌트에서 직접 만들지 않는다. 먼저 다음 공통 모델을 생성한다.

```json
{
  "schema_version": "marketplace_product_v1",
  "project_id": "uuid",
  "page_version_id": "uuid",
  "title": "상품명",
  "category": {
    "sellform_category": "living",
    "source_category_text": "생활/건강"
  },
  "brand": {
    "name": "브랜드명",
    "manufacturer": "제조사명"
  },
  "pricing": {
    "sale_price": 39900,
    "currency": "KRW",
    "stock_quantity": 100
  },
  "options": [],
  "delivery": {},
  "returns": {},
  "compliance": {},
  "assets": {
    "representative_images": [],
    "detail_page_artifacts": []
  },
  "evidence": {
    "verified_fact_ids": []
  }
}
```

공통 패키지는 생성 시점의 페이지 버전과 검증 사실을 참조한다. 이후 페이지가 바뀌어도 이미 승인된 등록 요청의 내용이 조용히 변경되지 않도록 불변 스냅샷을 만든다.

### 6.2 마켓별 변환기

변환기는 공통 패키지를 외부 API 요청 데이터로 바꾸는 순수 계층이다.

- `SmartStoreTransformer`: 네이버 카테고리, 원상품, 옵션, 배송, 고시, 이미지 필드로 변환
- `CoupangTransformer`: 쿠팡 카테고리, 고시정보, 옵션, 출고지·반품지, 이미지 필드로 변환

변환기는 HTTP 요청을 보내지 않는다. 외부 호출은 별도 어댑터가 담당한다.

### 6.3 마켓별 검증기

검증 결과는 다음 세 단계로 나눈다.

- `error`: 등록 전 반드시 수정해야 하는 항목
- `warning`: 등록은 가능하지만 사용자가 확인해야 하는 항목
- `ready`: 전송 가능한 상태

검증 항목에는 상품명, 카테고리, 가격, 재고, 옵션, 배송·반품 정보, 고시정보, 대표 이미지, 상세페이지 이미지 규격이 포함된다.

### 6.4 승인 게이트

승인 화면은 다음 내용을 한 번에 보여준다.

- 전송 대상 마켓과 연결된 판매자 계정
- 상품명, 카테고리, 가격, 재고, 옵션
- 대표 이미지와 상세페이지 미리보기
- 배송·반품·고시정보
- 경고와 자동 보정 내역
- 전송 이후 예상 상태

승인 시 서버는 패키지 해시, 승인 사용자, 승인 시각을 저장한다. 패키지가 변경되면 기존 승인은 무효가 되며 다시 승인해야 한다.

### 6.5 외부 API 어댑터

어댑터는 인증, 서명, 요청 전송, 응답 정규화만 담당한다.

- 네이버 어댑터: OAuth 2.0 토큰 처리 및 상품 API 호출
- 쿠팡 어댑터: API 키 기반 인증·서명 및 상품 API 호출

자격 증명은 데이터베이스에 평문으로 저장하지 않는다. 로컬 내부 도구에서는 환경 변수 또는 OS 비밀 저장소를 사용하고, 구독형 서비스에서는 암호화된 자격 증명 저장소로 교체한다.

## 7. 마켓별 고려사항

### 7.1 네이버 스마트스토어

네이버 커머스API 사용에는 커머스API센터 가입, 애플리케이션 등록 및 인증이 필요하다. 상품 등록 API에는 원상품, 카테고리, 옵션, 배송 등 여러 구조화 필드가 필요하다.

네이버 상품 등록 요청이 곧 외부 상태 변경을 일으킬 수 있으므로 Sellform의 `초안`은 네이버에 미리 전송하지 않는다. 사용자가 승인한 뒤에만 상품 등록 API를 호출한다.

공식 참고:

- [네이버 커머스API 소개](https://apicenter.commerce.naver.com/docs/introduction)
- [네이버 커머스 상품 등록 API](https://apicenter.commerce.naver.com/docs/commerce-api/current/create-product-product)

### 7.2 쿠팡

쿠팡 상품 생성에는 출고지·반품지, 카테고리, 고시정보, 옵션 등 마켓 규격에 맞는 값이 필요하다. 상품 생성 이후 승인 요청과 상태 확인이 별도 단계가 될 수 있으므로 Sellform 내부 상태도 `전송됨`, `승인 대기`, `등록 완료`, `수정 필요`, `실패`로 구분한다.

공식 참고:

- [쿠팡 상품 생성 API](https://developers.coupangcorp.com/hc/ko/articles/360033877853-%EC%83%81%ED%92%88-%EC%83%9D%EC%84%B1)
- [쿠팡 OPEN API 상품 등록 가이드](https://developers.coupangcorp.com/hc/ko/articles/360034889893-OPEN-API-%EC%83%81%ED%92%88-%EB%93%B1%EB%A1%9D-%EA%B0%80%EC%9D%B4%EB%93%9C)

## 8. 상태 모델

```text
draft
  → validating
  → needs_input | ready
  → awaiting_user_approval
  → submitting
  → submitted
  → approval_pending
  → published
```

실패 상태는 원래 상태를 잃지 않고 별도로 기록한다.

```text
validation_failed
authentication_failed
submission_failed
remote_rejected
status_check_failed
```

재시도는 동일한 멱등 키와 요청 스냅샷을 사용한다. 사용자가 내용을 수정했다면 새 요청 버전을 만들고 새 승인을 받아야 한다.

## 9. 데이터 모델

### `MarketplaceAccount`

- workspace 및 마켓 식별자
- 외부 판매자 계정 식별자
- 암호화된 자격 증명 참조
- 연결 상태와 마지막 검증 시각

### `MarketplaceProductPackage`

- 프로젝트 및 페이지 버전
- 공통 상품 데이터 JSON
- 패키지 해시와 생성 시각

### `MarketplaceListingDraft`

- 대상 마켓
- 변환된 요청 JSON
- 검증 결과
- 사용자 보정값
- 승인 상태

### `MarketplaceSubmission`

- 멱등 키
- 승인 사용자와 승인 시각
- 요청 스냅샷
- 외부 상품 식별자
- 현재 상태
- 마지막 오류

### `MarketplaceSubmissionEvent`

- 요청, 응답, 상태 변경, 재시도 이벤트
- 비밀값을 제거한 감사용 페이로드

## 10. UI 설계

상세페이지의 `저장/내보내기` 단계에 다음 동작을 추가한다.

1. `마켓 등록 준비`
2. `등록 패키지 다운로드`
3. `스마트스토어 연결`
4. `쿠팡 연결`

마켓 등록 화면은 다섯 구역으로 구성한다.

1. 마켓 및 계정 선택
2. 공통 상품 정보
3. 마켓별 필수정보와 자동 변환 결과
4. 이미지·상세페이지 미리보기
5. 오류·경고 및 최종 승인

`등록` 버튼은 `ready` 상태이며 사용자가 확인 체크박스를 선택한 경우에만 활성화한다.

## 11. 오류 처리 원칙

- 네트워크 오류와 입력값 오류를 구분한다.
- 외부 API의 원문 오류는 서버 로그에 보존하되 사용자에게는 수정 가능한 필드 중심으로 번역한다.
- 인증 실패 시 자동 반복 호출하지 않는다.
- 일시적 오류만 제한적으로 지수 백오프 재시도한다.
- 타임아웃이 발생해도 즉시 새 상품을 다시 만들지 않고 외부 등록 상태를 먼저 조회한다.
- 동일 요청의 중복 상품 생성을 막기 위해 멱등 키와 외부 상품 식별자를 저장한다.
- 비밀키, 토큰, 서명값은 로그와 API 응답에서 마스킹한다.

## 12. 테스트 전략

### 자동 테스트

- 공통 패키지 생성 및 불변 스냅샷
- 마켓별 변환기 계약 테스트
- 필수값 누락 및 규격 검증
- 승인 없는 전송 차단
- 패키지 변경 시 승인 무효화
- 멱등 요청 및 중복 방지
- 토큰 만료, 인증 실패, 요청 제한, 타임아웃
- 외부 오류 응답의 사용자 메시지 변환
- 자격 증명과 로그 마스킹

### 샌드박스·수동 테스트

- 테스트 판매자 계정 연결
- 대표 이미지와 상세페이지 이미지 확인
- 실제 등록 직전 미리보기 비교
- 등록 요청 후 외부 상태 조회
- 실패 수정 후 재시도
- 수동 등록 패키지 다운로드와 파일 구성 확인

실제 판매 계정으로 전송하는 테스트는 별도 환경 플래그와 명시적 사용자 승인 없이는 실행하지 않는다.

## 14. 완료 기준

- 스마트스토어와 쿠팡이 동일한 Sellform 공통 상품 패키지를 사용한다.
- 마켓별 변환 및 검증 로직이 UI와 외부 호출에서 분리되어 있다.
- 사용자 승인 전에는 외부 상품 생성 요청이 절대 실행되지 않는다.
- 승인한 요청의 스냅샷과 실제 전송 데이터가 일치한다.
- 오류가 어떤 필드를 어떻게 수정해야 하는지 사용자에게 표시된다.
- 재시도로 동일 상품이 중복 생성되지 않는다.
- API 연동을 사용하지 않는 사용자도 수동 등록 패키지를 받을 수 있다.
- 테스트 로그, 코드 리뷰, 트러블슈팅, 운영 런북이 각 Sprint 산출물로 남는다.

## 15. 향후 확장

- 상품 수정 및 판매 중지
- 추가 마켓 어댑터
- 팀 승인 워크플로
- 구독형 서비스용 다중 판매자 OAuth 및 자격 증명 보관
- 주문·배송·정산 연동
