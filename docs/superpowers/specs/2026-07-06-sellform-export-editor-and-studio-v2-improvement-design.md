## 1. 지금 실제로 실패하는 부분

현재 결과 화면은 겉으로 보면 업로드 가능한 상세페이지처럼 보이지만, 실제 프로젝트 데이터 기준으로는 아직 export 가능한 상태가 아니다.

확인한 실제 프로젝트:

- 프로젝트 ID: `fffe0d0f-64e0-4085-bcc4-239af755eaaf`
- 프로젝트명: `루메나 휴대용 무선 냉각선풍기`
- 확인 API: `GET /api/v1/projects/{project_id}/page/readiness`

현재 readiness 결과:

```json
{
  "ready": false,
  "blockers": [
    "asset_not_eligible",
    "visual_html_cards_required",
    "visual_html_cards_required",
    "asset_not_eligible",
    "visual_spec_rows_required"
  ]
}
```

따라서 PNG/JPG 다운로드가 안 되는 핵심 이유는 브라우저 저장 버튼 자체가 아니라, 백엔드가 “이 상세페이지는 아직 최종 이미지로 내보낼 준비가 안 됐다”고 차단하기 때문이다.

## 2. PNG/JPG 다운로드 실패의 원인

### 원인 A — 화면에 보이는 AI 생성 이미지가 export 가능 이미지로 인정되지 않음

현재 상세페이지는 AI 생성 이미지 2개를 사용한다.

- Hero 이미지 asset: `e1a59940-d632-4b17-bdbe-a52251214af2`
- Detail 이미지 asset: `a8ab869a-94fe-4097-9dd0-804dbf5637f5`

하지만 해당 프로젝트의 `image_generation_jobs` 기록이 비어 있다. 현재 export 정책은 다음 이미지만 허용한다.

- 사용자가 올린 이미지: `uploaded`
- 외부 출처 이미지: `sourced`
- 직접 촬영 이미지: `self_shot`
- 또는 `ImageGenerationJobRecord.status = approved`인 AI 생성 이미지

문제는 현재 AI 생성 이미지의 `source_type`은 `real-generated`인데, 승인된 생성 job 기록이 없다. 그래서 화면에는 이미지가 보이지만 export readiness에서는 “승인되지 않은 이미지”로 막힌다.

### 원인 B — HTML/CSS visual 섹션의 payload가 비어 있음

현재 일부 섹션은 이미지 생성 대신 HTML/CSS 그래픽으로 표현되도록 되어 있다. 방향 자체는 좋다. 문제는 payload가 완성되지 않았다는 점이다.

현재 데이터:

```json
{
  "comparison": { "layout_variant": "comparison_cards" },
  "detail_1": { "layout_variant": "benefit_cards" },
  "guarantee": { "layout_variant": "spec_table" }
}
```

필요한 데이터:

- `comparison_cards`에는 `cards`가 있어야 한다.
- `benefit_cards`에는 `cards`가 있어야 한다.
- `spec_table`에는 `table_rows`가 있어야 한다.

즉 “이 섹션은 카드형으로 보여줘”라는 이름만 있고, 실제 카드 내용이 없다.

### 원인 C — backfill 로직이 너무 얕음

현재 `backfill_page_visuals()`는 `visual_kind`가 이미 있으면 해당 섹션을 건너뛴다.

하지만 실제 문제 데이터는 이렇다.

- `visual_kind`는 있음
- `visual_payload`는 불완전함

따라서 backfill은 `visual_kind` 존재 여부가 아니라 `validate_visual()` 결과 기준으로 불완전한 payload를 보강해야 한다.

### 원인 D — 사용자에게 실패 이유가 충분히 보이지 않음

현재 프론트엔드에는 `exportError` 표시가 있지만, 사용자는 “다운로드가 안 된다” 정도로만 느낀다. 실제 blocker 목록이 사람이 이해할 수 있는 체크리스트로 표시되어야 한다.

예시:

```text
다운로드 전 확인이 필요합니다.

1. 생성 이미지 2개가 아직 검수 승인되지 않았습니다.
2. 비교 섹션의 카드 구성이 비어 있습니다.
3. 구매 전 확인 섹션의 표 정보가 비어 있습니다.

[검수하며 다듬기에서 해결하기]
```

## 3. 문구 품질 문제

현재 예시 문구:

```text
더위엔 ‘바람+냉각’—콘센트 없이 휴대해서 시원하게
```

이 문구가 어색한 이유:

- `바람+냉각`이 키워드 조합처럼 보인다.
- `—`가 한국 쇼핑몰 상세페이지 문맥에서 부자연스럽다.
- “콘센트 없이 휴대해서”도 말맛이 딱딱하다.

개선 방향:

```text
바람과 냉각감을 더해, 콘센트 없이도 시원하게
```

또는:

```text
무선으로 들고 다니며, 더운 순간 바로 시원하게
```

### 문구 생성/수정 규칙

AI 문구 생성과 AI 문구 수정에는 다음 규칙이 필요하다.

- `+`는 실제 제품명이나 브랜드 표기가 아니면 사용하지 않는다.
- `—`는 한국어 상세페이지 제목에서 사용하지 않는다.
- `[AI 수정됨]` 같은 내부 마커를 절대 출력하지 않는다.
- 검증되지 않은 최상급, 확정 표현, 수치 표현을 만들지 않는다.
- 제목은 구매자가 바로 이해할 수 있는 자연스러운 한국어로 만든다.

### 버튼별 기대 동작

| 버튼 | 기대 동작 |
| --- | --- |
| 제목 강화 | 제목만 더 구체적이고 강하게 재작성 |
| 짧고 자연스럽게 | 제목과 본문의 중복·군더더기 제거 |
| 과장 표현 축소 | 근거 없는 최상급·확정 표현 제거 |
| 사용 장면 보강 | 실제 사용 맥락을 본문에 자연스럽게 추가 |
| 초보 셀러 톤 | 쉬운 단어와 짧은 문장으로 변경 |
| 구매 불안 감소 | 확인된 정보와 구매 전 체크사항 추가 |
| 직접 요청 | 사용자의 자유 지시 반영 |

## 4. 이미지 생성 범위 문제

Sprint 64까지의 방향은 “모든 섹션을 이미지로 생성”이 아니라 이미지와 HTML/CSS 그래픽을 섞는 방식이다.

이 방향은 유지할 수 있다. 다만 사용자 입장에서는 HTML/CSS 그래픽 섹션이 충분히 디자인되어 있지 않으면 “이미지가 덜 만들어진 것”처럼 느껴진다.

따라서 앞으로는 모든 섹션이 아래 셋 중 하나로 완성되어야 한다.

| visual type | 의미 | 예시 |
| --- | --- | --- |
| `image_overlay` | 이미지 위에 HTML/CSS 문구를 얹음 | hero, 사용 장면 |
| `html_graphic` | 이미지 없이 HTML/CSS로 카드·표·비교 그래픽 구성 | 비교, 스펙, 체크리스트 |
| `source_photo` | 원본/업로드 이미지를 안전하게 crop하고 문구를 얹음 | 제품 실물 컷 |

핵심은 `html_graphic`이 “빈칸”이 아니라 “디자인된 그래픽”이어야 한다는 것이다.

## 5. 고급 편집기와 검수하며 다듬기 분리

현재 두 버튼은 사실상 같은 화면으로 연결된다.

```text
/workspace/projects/{project_id}/page-editor?mode=advanced
/workspace/projects/{project_id}/page-editor?mode=review
```

쿼리 값만 다르고, 실제로는 같은 `ReviewEditorLayout`을 사용한다. 그래서 사용자가 보기에는 “고급 편집기로 열기”와 “검수하며 다듬기”가 같은 페이지처럼 느껴진다.

### 검수하며 다듬기

목적:

- 판매자가 최종 출력 전 문제를 확인하고 해결한다.
- AI 문구 수정, 이미지 승인, export blocker 해결에 집중한다.

구성:

- 왼쪽: readiness 체크리스트와 섹션 목록
- 가운데: 구매자 화면 미리보기
- 오른쪽: 선택한 섹션의 빠른 수정/AI 수정/이미지 검수

### 고급 편집기

목적:

- 판매자가 상세페이지 구조와 시각 표현을 직접 조정한다.
- 검수보다 편집 자유도가 높아야 한다.

구성:

- 왼쪽: 섹션 트리, 순서 변경, 숨김/표시
- 가운데: 편집 가능한 canvas
- 오른쪽: 제목, 본문, 이미지, overlay, visual type 속성 패널

권장 구조:

```text
PageEditorShell
├─ ReviewPageEditor
│  ├─ ReadinessChecklist
│  ├─ BuyerPreview
│  └─ AiPolishPanel
└─ AdvancedPageEditor
   ├─ SectionTreePanel
   ├─ EditableCanvas
   └─ SectionPropertiesPanel
```

## 6. Hookable식 Studio V2 방향

첨부 영상에서 참고할 점은 “처음부터 완성 상세페이지를 보여주는 것”보다, 중간 단계의 `기획 초안` 경험이다.

영상 흐름에서 보인 강점:

- 상품명과 특징을 넣으면 먼저 여러 개의 기획 섹션이 생성된다.
- 사용자는 섹션을 보고 수정하거나 삭제할 수 있다.
- 그다음 상세페이지를 만든다.
- 최종 결과물은 이미지/연출컷처럼 다운로드 가능한 형태로 나온다.

Sellform도 바로 결과 화면으로 뛰기보다 아래 흐름을 갖는 편이 좋다.

```text
상품 입력
  ↓
AI 기획 초안 작성
  ↓
섹션별 검수/수정
  ↓
비주얼 전략 확정
  ↓
상세페이지 생성
  ↓
검수하며 다듬기
  ↓
PNG/JPG 다운로드
```

### 기획 초안 화면에 필요한 카드

- 히어로 메시지
- 타깃 고객
- 문제 상황
- 핵심 장점 3~5개
- 사용 장면
- 비교 포인트
- 구매 전 확인사항
- 구성품/스펙
- 주의사항
- 최종 CTA

이 단계가 있으면 사용자는 “결과물이 갑자기 이상하게 나왔다”고 느끼기 전에, 먼저 상세페이지의 논리와 구조를 고칠 수 있다.

## 7. 권장 Sprint 순서

### Sprint 65 — 실제 PNG/JPG 다운로드 통과

가장 먼저 해야 한다. 결과물이 좋아 보여도 다운로드가 안 되면 제품 약속이 깨진다.

다운로드 다음으로 중요하다. 현재 문구가 부자연스러우면 사용자는 결과물을 신뢰하지 못한다.

핵심:

- `+`, `—`, `[AI 수정됨]` 제거
- 버튼별 수정 의도 강화
- 수정 전/후 차이가 보이도록 preview 품질 강화

핵심:

- 상세페이지를 만들기 전에 구조를 먼저 보여준다.
- 사용자가 섹션을 수정/삭제/정렬한다.
- 승인된 기획안으로 최종 상세페이지를 생성한다.

## 9. 2026-07-07 추가 기획: 판매 카피·실상품 이미지·AI 버전 수정 고도화

Sprint 65~73을 진행하면서 확인된 다음 단계의 핵심 문제는 “기능 연결”보다 “결과물의 판매 품질”이다. Sellform의 최종 목표는 AI가 임의로 예쁜 상세페이지를 상상해서 만드는 것이 아니라, 상품 링크·상품 정보·업로드 이미지를 근거로 판매자가 바로 다듬어 업로드할 수 있는 고품질 상세페이지 초안을 만드는 것이다.

따라서 이후 개선은 아래 네 축으로 진행한다.

### 9.1 다운로드와 출력 이력은 하나의 계약으로 정리한다

PNG/JPG 저장은 다음 조건을 만족해야 한다.

- 상세페이지 본문 영역만 저장한다.
- 상단 네비게이션, 검수 패널, 이미지 후보 패널, 저장 버튼은 캡처에 포함하지 않는다.
- 파일명은 UUID가 아니라 상품명 기반으로 생성한다.
  - 예: `루메나-휴대용-무선-냉각선풍기-상세페이지.jpg`
- 기본 동작은 브라우저 일반 다운로드로 통일해 Chrome 다운로드 기록에 남긴다.
- `showSaveFilePicker` 기반 “다른 이름으로 저장”은 별도 선택 기능으로만 둔다.
- 다운로드 완료 후 같은 export artifact가 출력 이력에 기록되고, 출력 이력에서 다시 받을 수 있어야 한다.

### 9.2 대본은 내부 기획 문구가 아니라 고객 설득 카피여야 한다

현재 일부 결과물에는 AI에게 줄 지시문이 고객용 문구처럼 노출된다.

금지해야 할 예:

- `가장 먼저 보여줄 핵심 사용 가치를 한 문장으로 정리합니다.`
- `상품 입력 정보를 바탕으로 안전한 표현을 사용합니다.`
- `구매자가 불편을 느끼는 순간부터 보여주세요.`
- `확인된 장점만 또렷하게 정리해요.`

최종 상세페이지에는 구매자가 읽는 문장만 들어가야 한다. AI는 상품 링크 또는 입력 정보에서 실제 판매 포인트를 추출하고, 다음 흐름으로 대본을 구성한다.

1. 문제 제기: 고객의 핵심 불편을 먼저 보여준다.
2. 메인 소구점: 이 제품이 핵심 문제를 어떻게 해결하는지 말한다.
3. 소구점 A: 가장 강한 장점을 구체화한다.
4. 소구점 B: 추가 장점을 사용 장면과 연결한다.
5. 소구점 A 재강조: 핵심 메시지를 다른 표현으로 다시 강조한다.
6. 소구점 B~D 정리: 나머지 장점을 카드·리스트로 정리한다.
7. 전체 요약: 구매 이유를 한 문장으로 정리한다.
8. 상품 정보: 스펙, 구성품, 충전 방식, 사용 조건, 구매 전 체크사항을 정리한다.

대본 품질 기준:

- 상품명, 상품 링크, 입력 특징, 업로드 이미지에서 확인 가능한 정보만 사용한다.
- 없는 기능, 수치, 인증, 효과를 만들지 않는다.
- `+`, `—`, `[AI 수정됨]` 같은 어색한 마커를 쓰지 않는다.
- “정리합니다”, “보여주세요”, “입력 정보를 바탕으로” 같은 내부 표현을 최종 문구에 남기지 않는다.
- 제목은 `무엇을`, `어디서`, `왜 쓰는지`가 바로 보이게 쓴다.

### 9.3 사용자는 수기로 수정하고, AI는 버전별 수정안을 제안한다

판매자는 AI 초안을 그대로 쓰지 않을 수 있다. 따라서 모든 섹션의 제목과 본문은 직접 수정 가능해야 하며, AI 수정은 현재 문구를 기준으로 “대체안”을 제안해야 한다.

AI 수정 버튼 예:

- 강한 구매 설득 버전
- 짧고 임팩트 있는 버전
- 초보 셀러 자연스러운 버전
- 프리미엄 브랜드 톤
- 쿠팡/스마트스토어 최적화 버전
- 과장 줄인 신뢰형 버전
- 감성 라이프스타일 버전
- 구매 불안 감소 버전

버튼을 누르면 즉시 덮어쓰지 않고 비교 모달을 보여준다.

```text
수정 전:
냉각선풍기, 필요한 순간 바로 쓰는 선택

수정 후:
책상·차량·야외까지, 더운 순간 바로 꺼내 쓰는 무선 냉각 선풍기

[이 수정안 적용] [다시 생성] [취소]
```

수기 수정한 문구는 보호되어야 한다. AI는 사용자가 이미 고친 내용을 무시하고 처음부터 다시 쓰는 것이 아니라, 현재 편집본을 기반으로 더 나은 버전을 제안한다.

### 9.4 이미지는 실제 상품 누끼를 중심으로 장면에 합성한다

AI가 상품 본체를 새로 그리면 실제 상품과 달라질 수 있다. 팬 모양, 버튼 위치, 색상, 로고, 비율이 바뀌는 문제가 생긴다. 따라서 상품 본체는 업로드된 실제 상품 이미지에서 누끼를 따서 유지하고, AI는 배경·조명·그림자·사용 장면을 보정하는 역할을 맡는다.

이미지 모드는 세 가지로 나눈다.

| 모드 | 설명 | 사용 위치 |
| --- | --- | --- |
| 정확도 우선 | 실제 상품 누끼를 거의 그대로 사용 | Hero, 스펙, 구매 전 체크 |
| 자연스러운 합성 | 누끼 상품에 배경, 그림자, 조명을 맞춤 | 라이프스타일, 책상, 침대, 차량, 야외 |
| AI 보정 후보 | 손에 든 장면, 측면 느낌 등 AI가 일부 변형 | 사용 장면 후보, 검수 필요 |

정면 사진 1장만으로 손에 든 장면이나 측면 장면을 만들면 AI가 보이지 않는 부분을 추정해야 한다. 이 경우 상품이 달라질 위험이 있으므로 결과 후보에 `AI 보정 후보 - 검수 필요` 라벨을 표시한다.

섹션별 렌더링 원칙:

| 섹션 | 처리 방식 |
| --- | --- |
| Hero | 실제 상품 누끼 + 고급 배경 + HTML 문구 |
| Problem | 문제 상황 이미지 또는 HTML 카드 |
| Lifestyle | 누끼 상품을 장면에 합성 |
| Comparison | HTML/CSS 비교 카드 + 상품 누끼 |
| Features | 아이콘/카드형 HTML 그래픽 |
| Specs | 이미지 생성하지 않고 HTML/CSS 표 |
| Pre-purchase | 체크리스트 카드 |
| CTA | 상품 누끼 + 마지막 확인 문구 |

`graphic_chart` 섹션은 사진 생성이 아니라 HTML/CSS 그래픽으로 처리한다. 스펙, 비교, 구매 전 체크처럼 정확성이 중요한 정보는 AI 이미지보다 구조화된 HTML이 더 안전하다.
